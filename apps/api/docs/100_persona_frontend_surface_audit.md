## 0. Project context (read first)

**Assembly** is a synthetic-society simulation lab for founder product briefs. A founder submits a brief (product, price, target customers, competitors, geography). Assembly retrieves live evidence from the web, builds a synthetic society of evidence-anchored personas, runs a multi-round debate, and returns a report describing how the market would react. The product is live at `https://run.assemblysimulator.com/`.

**Calibration-first product principle**: no numeric forecasts, no forced verdicts, no fake data, honest confidence bands. The simulation is explicitly framed as a synthetic signal, not a real-world purchase forecast.

**Stack**:
- **Backend (Python / FastAPI)**: orchestrates the simulation, calls LLMs (Anthropic Sonnet for role-play, Opus for synthesis), runs retrieval (Brave, Tavily, YouTube, Firecrawl, Amazon Reviews 2023), persists to PostgreSQL.
- **Frontend (Next.js 14 + React + Tailwind)**: founder-facing UI at `run.assemblysimulator.com`. Hosted on Vercel.
- **Deploy**: API on Railway, web on Vercel. Both auto-deploy on `main` push.

## 1. Current architectural truth (this is the part everyone misreads)

**There are already two layers of persona in every simulation, not one.**

| Layer | Count | What they do | Cost per persona |
|---|---|---|---|
| **Deep agents** (rich personas) | 21–30 (founder picks within this range; default 24) | Full LLM debates: 4 groups × 4 rounds × ~6 turns each, with pre-ballot, reflection, final-ballot stages. Speak, react, push back, change minds. These are what you see in the on-screen **Debate transcript** and the **Society composition** card. | High — each persona makes multiple LLM calls per round, growing roughly quadratically with population because debate-pair sampling scales with N. |
| **Lightweight voters** | 100 (fixed, hardcoded in `lightweight_voter_pipeline.py` as `_VOTER_COUNT = 100`) | Derived from the rich-persona cohort centroids. Run a **4-round influence loop** over cluster-level arguments extracted from the deep-agent debate. **No new LLM calls per voter.** | Effectively zero — voters are deterministic samples from cohort posteriors. |

### Why both, not just 100 deep agents

This was a deliberate Phase 12C decision after empirical validation:

1. **Variance starvation above ~30 deep agents.** When we scaled deep agents past 30, the diversity gates failed: stdev across runs ≤5pp, persona-to-persona TVD ≤0.12. More deep agents started agreeing with each other, so we added cost without adding signal.
2. **Cost scales quadratically.** 100 deep agents ≈ 10× the LLM bill of 24 deep agents because debate sampling is pairwise.
3. **Distribution shape signal is what 100 of anything is for.** The thing 100-persona populations are good at is producing tight distribution shapes (intent buckets, stance shares, cohort weights). That's exactly what the voter overlay produces — at near-zero cost, via cohort-centroid sampling + influence dynamics, not via 100 independent LLM debates.
4. **It works.** The voter overlay is already shipped (Phase 12C, ~50 lines of `lightweight_voter_pipeline.py`, ~9 supporting modules in `apps/api/src/assembly/sources/lightweight_voters/`). It runs on every simulation in production. It writes a `lightweight_voters.json` artifact to the run directory. It feeds into the simulated-intent layer that already appears in the report.

### How the 100 voters "talk" (this is the critical question the user is asking)

**They do not have LLM-style conversations with each other.** That would defeat the cost design and re-introduce the variance starvation problem at 100×.

Instead, voters interact through a **4-round bounded-confidence influence loop** ([Hegselmann–Krause model](https://en.wikipedia.org/wiki/Bounded_confidence_model), adapted):

1. **Round 0 (pre)**: Each voter is initialized with a stance/intent vector sampled from its parent cohort's centroid. Voters carry a behavioral signature (susceptibility to peer shift, influence weight, etc.) inherited from the cohort.

2. **Cluster arguments**: From the rich-persona debate, the orchestrator extracts the strongest pro/con/proof/objection arguments per cohort. These become the "discourse" that voters react to. The arguments themselves were generated by deep agents talking — voters are downstream of that.

3. **Rounds 1–4 (influence loop)**: Each voter:
   - Receives a sampled subset of cluster arguments weighted by social-graph proximity
   - Updates its stance vector if a counter-argument crosses its trust threshold AND the source voter falls inside the bounded-confidence radius
   - Otherwise holds
   - **No free-text generation. No LLM. Pure deterministic update rules.**

4. **Final aggregation**: After round 4, voter intent distribution is computed. This is the "100-voter signal" that should be surfaced to the user.

**Plain-English answer to "do they talk in groups or each with each other?"**: Neither, exactly. They influence each other through a structured 4-round loop that propagates the *deep agents'* arguments through a 100-node social graph. Voters change their mind in response to peer voters' stance shifts, but they don't generate new text. The deep agents do the talking; voters do the spreading.

### What the founder currently sees on the simulation page

| Element | Shows | Source |
|---|---|---|
| "Number of agents" slider on the brief form | 21–30, default 24 | Deep-agent count only |
| Society composition card | 24 personas, role makeup | Deep agents only |
| Debate transcript | 4 groups × 4 rounds, ~96 turns | Deep agents only |
| Intent distribution chart | Bucketed intent counts | **Already includes voter signal feeding in, but unattributed and indistinguishable from deep-agent intent** |
| Cohort cards | Cohort centroids + sizes | Derived from deep agents, voters not surfaced |
| Downloaded HTML/PDF report | All of the above | Same — voters absent |

**The 100 voters do their work silently and feed the final intent distribution, but the founder has no way to know they exist.** That's the gap.

## 2. What the user (founder) wants

Direct quotes from the operator (paraphrased):

> "I want us to show the user that there are now 100 personas as well, and now that the user can choose all the way from 30 to 100 personas to spawn. And then a big graph showing 100% of personas as being shown as well comes up — and it's not only a graph which looks good, but it actually makes sense as well. The same way we are showing a graph right now of 21 personas to 30 personas."

> "How are we going to implement the 100 personas to our frontend, and how are we going to make them talk — are we going to make them talk in groups or each one is going to talk with each other?"

### Interpreted into specific deliverables

1. **Surface the 100-voter layer in the UI** so founders see that the simulation runs a bigger sample than the 21–30 deep-agent number suggests.
2. **Brief-form slider that goes 30 → 100** (currently 21 → 30).
3. **A graph that visualizes the 100-voter signal** with the same visual quality as the existing intent / society composition graphs.
4. **Honest explanation in the UI** about what the 100 voters are doing — including the "do they talk" question.

### The conflation we must NOT pass through silently

The user is asking for "30 to 100 personas" as if they're all the same thing. But the architecture splits them — and for good reason (Section 1). ChatGPT needs to help decide how the UI slider should map to the underlying two-layer model.

**Three candidate UX models** (this is where ChatGPT's input matters):

### Model A — Two separate sliders

Show the founder two controls:
- **Debate depth** (deep agents): 21–30, default 24
- **Voter sample size** (lightweight voters): 50–200, default 100

Honest about the architecture. Slightly more cognitive load. Founders who don't care can leave both at defaults.

### Model B — One slider, smart split

Show the founder one slider: **Society size: 30 to 100**. Backend maps this to (deep_count, voter_count) automatically:
- 30 personas → 24 deep + 6 voters? Or 24 deep + 30 voters? (decision needed)
- 60 personas → 24 deep + 36 voters
- 100 personas → 24 deep + 76 voters (or fix deep at 24 and let voter count be `size - 24`)

Or alternative split: deep count grows slowly with the slider (24 → 30) while voter count fills the rest.

Lower cognitive load. Hides the architectural truth from the founder. Risk: when the founder picks 100 and gets a report based on 24 deep + 76 voters, are they being misled?

### Model C — One slider for *deep* depth, voters always 100

Show the founder one control: **Debate depth: 21 to 30** (current behavior, unchanged). Add a small UI panel that always says "Plus 100 lightweight voters that propagate the debate" — non-configurable, just informational. The "100" is always there; the founder can only tune debate depth.

Lowest UX risk. Honors the architecture. Doesn't satisfy the literal "slider from 30 to 100" ask, but it honestly surfaces the 100-voter layer that already exists.

**My (Claude's) instinct is Model C is the right calibration-first answer.** Model A is acceptable. Model B risks misleading founders about what "100 personas" means.

ChatGPT — your job: weigh in. Is there a reason to prefer A or B over C that I'm missing? Is there a fourth model worth considering?

## 3. What's already shipped (backend) — do NOT redo this

These files exist in production on `origin/main`:

```
apps/api/src/assembly/pipeline/lightweight_voter_pipeline.py
   _VOTER_COUNT = 100
   def run_lightweight_voter_overlay(...) -> dict[str, Any]:
       # 1. Derive cluster arguments from rich-persona ballots
       # 2. Generate 100 voters from cohort centroids
       # 3. Build social graph
       # 4. Run 4-round influence loop
       # 5. Aggregate voter distribution
       # 6. Write lightweight_voters.json to run_dir

apps/api/src/assembly/sources/lightweight_voters/
   voter_schema.py        — Voter, VoterCohort, VoterSocialGraph types
   voter_sampling.py      — generate_voters_from_cohorts(...)
   social_graph.py        — build_social_graph(...)
   influence_loop.py      — run_influence_rounds(...)
   aggregation.py         — voter intent aggregation
   calibration_correction.py — voter distribution calibration
   diversity_health.py    — voter-pool diversity diagnostics
```

Wired in: `apps/api/src/assembly/orchestration/live_founder_brief.py:2780` calls `run_lightweight_voter_overlay(...)` after the deep-agent debate completes.

Artifact: every simulation writes `lightweight_voters.json` to its run directory. The structure includes:
- `voters`: 100 voter records with cohort_id, social_graph_neighbors, behavioral_signature, intent_vector_over_time
- `influence_rounds`: round-by-round shift records
- `final_distribution`: aggregate intent breakdown after round 4
- `cluster_arguments`: which arguments propagated, which were resisted
- `samples`: 5–10 representative voter narratives (paraphrased; not LLM-generated, derived from cohort centroids)

**There is no API endpoint that serves `lightweight_voters.json` to the frontend yet.** That's the first concrete missing piece.

## 5. Graph design — the heart of this work

The user explicitly said: *"a graph showing 100% [of personas] as being shown as well, and it's not only a graph which looks good, but it actually makes sense."*

Three candidate graph designs. ChatGPT — please assess which is most "looks good AND makes sense."

### Graph 1 — Voter intent distribution (bar chart, ordered)

100 voters bucketed into intent categories: `would_buy_now`, `would_try_once`, `would_consider_if_proven`, `would_compare_to_current_brand`, `loyal_to_current_alternative`, `would_reject`. Horizontal stacked bar OR vertical bars sorted by count.

Pros: Most readable. Matches the existing intent distribution viz the user already shows for deep agents — natural extension.
Cons: Loses the "they influence each other" story.

### Graph 2 — Stance migration across 4 influence rounds (sankey / parallel categories)

Show each voter's stance at pre → round 1 → round 2 → round 3 → round 4. A sankey diagram where the ribbons widen or narrow as voters change their mind. Reveals which arguments moved which subgroups.

Pros: Visually striking. Tells the influence story directly.
Cons: Harder to read at a glance. Sankey libraries are heavier. May require fewer buckets to stay legible.

### Graph 3 — Cohort centroid landscape + voter cloud

Plot 100 voters as dots in a 2D embedding (intent strength × persuadability, or similar derived axes). Color by cohort. Overlay the rich-persona cohort centroids as labeled markers. The "spread" of voters around each centroid visualizes within-cohort variance.

Pros: Most "AI-lab demo" looking. Surfaces the cohort structure explicitly.
Cons: Risks looking like fake-precision dimensionality reduction unless the axes have clear meaning.

### My (Claude's) preference

Graph 1 (intent distribution bar chart) as the primary viz — same shape as the existing deep-agent chart, easy to compare, honest. Optionally add Graph 2 (stance migration sankey) as an "expand for influence dynamics" secondary panel.

Avoid Graph 3 unless we can defend the axes — derived 2D projections can leak fake-precision into a calibration-first product.

- Which of these honors the calibration-first principle best?
- Is there a fourth graph I'm not considering?
- Should the voter chart and the deep-agent chart be **side-by-side** (so the founder sees agreement / disagreement between layers) or **stacked** (voter chart augments the existing deep-agent chart)?

## 6. The "do they talk" question — proposed UI explanation

When founders see a 100-voter section in the UI, they will ask: *"Wait — are these talking to each other? Are they real LLM conversations?"*

We must answer honestly. Proposed UI copy block:

> **How the 100 voters work**
>
> The 24 personas in the **Debate transcript** above are doing the actual talking — they're full LLM agents that argue, push back, and change each other's minds across 4 groups × 4 rounds.
>
> The 100 voters below are a **larger sample** drawn from the same evidence the 24 personas were built from. They don't write text. Instead, they react to the arguments the 24 personas made — propagating the strongest pro/proof/objection points through a 100-voter network over 4 influence rounds.
>
> Think of it as: **24 personas debate; 100 voters absorb and spread**.
>
> This is why the 100-voter distribution can differ from the 24-persona distribution: voters that get exposed to a strong proof argument shift their stance; voters insulated from it don't. The 100-voter signal tells you about the *reach* of an argument, not just its quality.

ChatGPT: critique the copy. Is it accurate? Founder-readable? Does it overpromise or underpromise?

## 9. Things to NOT do (anti-goals)

- **Do not scale deep agents to 100.** The Phase 12C decision rejected this on variance + cost grounds and the data is documented. The voter layer exists *because* we already tried this.
- **Do not pretend 100 voters are "the same as" 100 deep agents.** They're a different signal: distribution shape and influence reach, not qualitative argumentation. The UI copy must keep these distinct.
- **Do not introduce free-text LLM generation per voter.** That's the path that blew up the cost budget at scale.
- **Do not raise the strict calibration gates.** Anti-overfit guardrails stay locked.
- **Do not add Phase 13 / behavioral_mind_layer files to this work.** Those stay local-only on the operator's machine; production behavior is unchanged.

## 10. Risks worth flagging

- **Cognitive overload risk**: adding more sliders / charts can confuse founders who currently submit a brief and get a report. Mitigation: visible-by-default voter panel, but slider stays single-knob (Model C) unless ChatGPT argues otherwise.
- **Misleading number risk**: if a founder sees "100 personas" they may believe they got 100 LLM-personas worth of qualitative depth. Mitigation: clear UI copy (Section 6) about the two layers.
- **Performance risk**: rendering a 100-voter graph (especially a sankey) can be slow if implemented naively. Mitigation: use static SVG with pre-aggregated buckets; voters aren't rendered as individual DOM nodes.
- **PDF page-break risk**: adding a new section to the downloaded report can push the existing layout. Mitigation: explicit page break before the voter section.
- **Voter calibration risk**: the influence loop is calibrated for N=100. Allowing 50 or 200 may need re-calibration. Mitigation: validate before exposing the slider; default to 100 even if configurable.
